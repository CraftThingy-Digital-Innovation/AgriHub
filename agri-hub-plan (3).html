2
Matching Demand-Supply Antarwilayah
Mencocokkan surplus panen di satu daerah dengan permintaan di daerah lain. Realisasi: petani/koperasi lapor stok via WA → sistem cari pembeli yang butuh komoditas itu → notifikasi kedua pihak → mereka negosiasi & deal sendiri. Platform sebagai penghubung, bukan perantara transaksi.
💡 Cara Realisasi yang Masuk Akal
Supply side: Petani/koperasi lapor stok — komoditas, jumlah, harga, lokasi, tanggal tersedia → masuk "papan stok aktif".
Demand side: Distributor/pedagang besar lapor kebutuhan — komoditas, jumlah, harga maks, kota tujuan, deadline → masuk "papan kebutuhan".
Matching engine: Tiap ada entri baru, sistem query DB cari match: komoditas sama + harga dalam ±20% + jarak ≤500km + tanggal ±7 hari. Jika cocok → notifikasi WA ke kedua pihak.
Kontak: Pihak tertarik balas HUBUNGI → platform kirim nomor WA masing-masing untuk negosiasi langsung.

03 — Pembayaran & Keuangan

Midtrans — Pembayaran, Escrow & Laporan Keuangan

Semua transaksi marketplace dan matching dilindungi oleh sistem escrow berbasis Midtrans. Dana ditahan hingga transaksi selesai, melindungi buyer dan seller dari penipuan. Laporan keuangan, penarikan dana, dan cek saldo bisa dilakukan langsung via WhatsApp.

🛡️ Kenapa Midtrans? — Anti-Penipuan & Ajuan Banding

Midtrans adalah payment gateway Indonesia yang sudah berizin OJK. Dana buyer ditahan di escrow Midtrans — seller tidak bisa menerima uang sebelum buyer konfirmasi barang diterima. Jika terjadi sengketa, buyer atau seller bisa ajukan banding (BANDING via WA) → admin platform review → keputusan dieksekusi melalui Midtrans Refund API. Tidak ada uang yang bisa "kabur" karena semua ditahan di level payment gateway.

🔄 Alur Escrow — Dari Bayar Hingga Cair
1
Buyer buat pesanan → terima link Midtrans via WA Setelah BELI dikonfirmasi, server generate Midtrans transaction via POST /v2/charge. Link payment dikirim ke WA buyer. Mendukung: QRIS, GoPay, OVO, Dana, VA BCA/Mandiri/BNI/BRI, kartu kredit.
2
Buyer bayar → dana masuk escrow Midtrans Midtrans webhook notifikasi server (payment_notification). Status order berubah jadi "dibayar". Seller dinotifikasi WA: "✅ Pesanan #1234 sudah dibayar — siapkan pengiriman."
3
Seller konfirmasi kirim (via WA atau website) Seller kirim: KONFIRMASI 1234 + nomor resi. Buyer dinotifikasi: "📦 Pesanan dikirim. Resi: SCLB123. Konfirmasi terima dalam 3 hari atau otomatis diterima."
4
Buyer konfirmasi terima / auto-release 3 hari Buyer kirim TERIMA 1234 → dana dilepas ke saldo seller (minus fee platform). Jika tidak ada respons dalam 3 hari sejak resi terdeteksi sampai → otomatis release. Ini mencegah seller ditahan terus pembayarannya.
!
Sengketa → Ajuan Banding (window 3 hari setelah pengiriman) Buyer kirim BANDING 1234 [alasan] → status order jadi "sengketa" → dana di-freeze. Admin platform review (bukti foto/resi) → keputusan: refund buyer via Midtrans atau release ke seller. Keputusan final dalam 3x24 jam.
💰 Struktur Biaya — Shared Cost Model

Biaya admin platform dihitung dari pool seluruh pendapatan platform dibagi rata ke semua user aktif. Makin banyak user dan transaksi, makin murah biaya per user — karena server cost ditanggung bersama.

🖥️ Shared Server Cost Pool
Total biaya server/bulan dibagi ke semua user aktif. Contoh: server Rp500.000/bulan dengan 1.000 user aktif = Rp500/user. Dengan 10.000 user = hanya Rp50/user. Biaya makin murah seiring pertumbuhan platform. Pool ini diambil dari platform fee, bukan ditagih langsung ke user.
📊 Contoh Perhitungan — Transaksi Rp500.000
Nilai transaksi buyer bayar : Rp500.000
Midtrans MDR (~0.7% QRIS) : Rp3.500 ← ditanggung buyer (sudah include di total)
Platform fee 2% : Rp10.000 ← dipotong dari penerimaan seller
PPN 11% × platform fee : Rp1.100 ← platform setor ke kas negara
Seller terima bersih : Rp488.900
Platform revenue (sebelum server) : Rp8.900 (Rp10.000 - Rp1.100 PPN)

// payment/midtrans.ts — buat transaksi + kirim link ke WA
import midtransClient from 'midtrans-client'
const snap = new midtransClient.Snap({ isProduction: true, serverKey: process.env.MIDTRANS_SERVER_KEY })

export async function createPayment(order: Order, buyer: User) {
  const txn = await snap.createTransaction({
    transaction_details: { order_id: order.id, gross_amount: order.total },
    customer_details: { first_name: buyer.name, phone: buyer.phone_wa },
    expiry: { duration: 24, unit: 'hours' }
  })
  // Kirim link ke WA buyer
  await sendWA(buyer.phone_wa,
    `💳 Bayar pesanan #${order.id} (Rp${order.total.toLocaleString('id')})\n\nLink: ${txn.redirect_url}\n\nBerlaku 24 jam. Balas BANDING ${order.id} jika ada masalah.`)
}

// Webhook handler — dipanggil Midtrans saat status berubah
app.post('/api/payment/webhook', async (req, res) => {
  const notif = await snap.transaction.notification(req.body)
  if (notif.transaction_status === 'settlement') {
    await markOrderPaid(notif.order_id)
    await notifySellerWA(`✅ Pesanan #${notif.order_id} DIBAYAR. Siapkan pengiriman!`)
  }
})

06 — Chat Feature

Upload Dokumen Personal ke Chatbot

User bisa upload dokumen milik mereka sendiri langsung di halaman chat — AI membaca isinya dan bisa menjawab pertanyaan tentang dokumen itu. Berbeda dari RAG global: dokumen ini bersifat personal dan tidak masuk knowledge base publik.

📄 RAG Global (Admin)
Dokumen diupload admin → di-chunk → di-embed → masuk VectorDB → jadi pengetahuan semua user. Proses async, bisa memakan beberapa menit. Permanen.
📎 Upload Personal (User)
User upload dokumen di chat → isi di-parse langsung → dimasukkan ke context window AI → AI bisa langsung jawab. Proses instant, hanya untuk sesi user itu. Tidak disimpan permanen.
1
User klik ikon 📎 di interface chat, pilih file Format yang didukung: PDF, DOCX, XLSX, TXT, JPG/PNG (gambar tabel/brosur). Batas ukuran: 10 MB per file, maks 3 file per sesi. File diupload ke endpoint /api/chat/upload-doc.
2
Server parse dokumen jadi teks PDF → pdf-parse, DOCX → mammoth, XLSX → xlsx, gambar → Puter Vision AI (OCR). Teks hasil parse disimpan sementara di memori server (atau Redis TTL 1 jam), tidak ke DB permanen.
3
Teks dokumen diinjeksi ke context prompt AI Karena model default (Qwen3.5 Flash) mendukung 1M token context, seluruh isi dokumen langsung dikirim sebagai bagian dari prompt — tidak perlu chunking/embedding. Untuk dokumen besar (>50 halaman), baru dilakukan chunking sederhana.
4
AI merespons berdasarkan isi dokumen + knowledge RAG global Contoh: petani upload brosur pupuk NPK baru → tanya "ini cocok buat tanaman cabai saya nggak?" → AI baca brosur + ambil data dari RAG global tentang kebutuhan pupuk cabai → jawab spesifik dan relevan.
5
Dokumen tersedia selama sesi — user bisa terus bertanya Dalam satu sesi chat, user bisa ajukan banyak pertanyaan tentang dokumen yang sama. Sesi berakhir jika user tutup tab atau tidak aktif 60 menit. Dokumen tidak disimpan setelah sesi berakhir kecuali user klik "Simpan ke Koleksi Saya".

11 — WhatsApp Group

WhatsApp Bot di Grup — Group Owner Credits

Bot bisa diundang ke grup WhatsApp. Semua biaya AI di grup tersebut ditanggung oleh user yang mengundang bot (group owner), bukan anggota lain.

💡 Konsep Utama: Group Owner = Penanggung Credits

Saat bot diundang ke grup oleh User A, semua query AI dari siapapun di grup tersebut akan dieksekusi menggunakan Puter token milik User A. Anggota grup lain tidak perlu punya akun Puter sama sekali — mereka hanya "numpang" credits si pengundang.

1
Bot diundang ke grup oleh User A Baileys trigger event group-participants.update dengan action add. Sistem catat: grup ID + nomor WA User A sebagai "group owner" di tabel bot_groups. Bot langsung kirim pesan perkenalan.
2
Bot kirim pesan perkenalan ke grup Contoh pesan: "Halo semua! 👋 Saya AgriBot, asisten pertanian AI. Mention saya dengan @AgriBot untuk bertanya. Catatan: biaya AI ditanggung oleh @UserA yang mengundang saya. Ketik @AgriBot bantuan untuk melihat perintah."
3
Anggota grup kirim pesan dengan @mention atau reply bot Bot hanya merespons jika di-mention (@AgriBot) atau ada reply ke pesan bot. Pesan tanpa mention diabaikan — ini penting agar credits tidak boros.
4
Sistem eksekusi AI pakai token milik group owner Lookup bot_groups → dapat user ID group owner → ambil Puter token mereka → jalankan RAG + AI call. Usage dicatat ke tabel credit_usage dengan tag source=group, group_id, dan triggered_by (nomor WA yang bertanya).
5
Daily limit per grup tercapai Jika grup sudah memakai kuota harian (default: 30 query/hari), bot balas: "Kuota harian grup ini sudah habis (30/30). Owner grup dapat melihat penggunaan di agrihub.id/akun atau ketik @AgriBot cek kredit."
6
Group owner hapus akun / cabut koneksi Puter Sistem deteksi token tidak valid → kirim pesan ke semua grup milik owner tersebut → bot otomatis keluar (sock.groupLeave(groupId)). Anggota grup diberitahu dan disarankan cari owner baru.

sock.ev.on('group-participants.update', async ({ id, participants, action }) => {
  const botJid = sock.user.id

export async function handleCreditCheck(user: User, args: string) {
  const now    = new Date()
  const period = args.includes('hari ini') ? 'today'
               : args.includes('30 hari')  ? '30d'
               : 'month'

  const usage = await db('credit_usage')
    .where({ user_id: user.id })
    .where('created_at', '>=', getPeriodStart(period))
    .select(
      db.raw('source, SUM(tokens_in + tokens_out) as total_tokens, SUM(est_cost_usd) as total_cost')

    .groupBy('source')

  const groups = await db('bot_groups')
    .where({ owner_user_id: user.id, is_active: true })
    .select('group_name', 'daily_used', 'daily_limit')

  return formatCreditReport(usage, groups)  // format jadi teks WA-friendly
}

// ai/modelRouter.ts — pilih model berdasarkan konteks
export function selectModel(opts: {
  queryType: 'simple' | 'rag' | 'multi_doc',
  creditUsedPct: number  // 0–1
}): string {
  // Safety net: kalau sudah >80% budget bulan ini → model gratis
  if (opts.creditUsedPct > 0.80)
    return 'arcee-ai/trinity-large-preview'  // FREE

  // RAG multi-dokumen → 235B, output token termurah
  if (opts.queryType === 'multi_doc')
    return 'qwen/qwen3-235b-a22b-2507'  // $0.07/$0.10

  // Default semua query → Qwen3.5 Flash
  return 'qwen/qwen3.5-flash-02-23'  // $0.07/$0.26
}

JetBrains Mono — Data & kode
Untuk harga, angka statistik, dan UI teknis
✨ Animasi — Framer Motion
Page transition fade-up, stagger animation kartu tanaman, scroll-triggered reveal untuk statistik harga. Spring physics untuk sidebar dan modal.
📱 Mobile-First
Layout bottom-nav untuk mobile. Ukuran teks minimal 16px. Tombol besar minimal 48px tap area. Petani sering akses dari HP android mid-range.
♿ Aksesibilitas
WCAG 2.1 AA contrast ratio. Semua teks penting bisa diperbesar tanpa layout rusak. Voice input support untuk chat AI (Web Speech API).
🌙 Dark Mode
Dark mode otomatis ikuti sistem OS. Palet dark: deep forest green (#0d2010) background, lighter greens untuk surface, amber tetap sebagai accent.
🇮🇩 Lokalisasi
Bahasa Indonesia penuh. Format angka Rp (rupiah). Nama tanaman dalam Bahasa Indonesia + nama latin opsional. Kalender Jawa untuk musim tanam opsional.
⚡ Performance
Core Web Vitals target: LCP <2.5s, CLS <0.1. Lazy load gambar, infinite scroll daftar tanaman, SWR cache untuk data harga.

15 — Roadmap

Roadmap Sprint Development

Estimasi 4 sprint masing-masing 2 minggu. Solo developer atau tim kecil 2-3 orang.

S1
Sprint 1 — Minggu 1–2
Foundation & Core Backend
  • Setup monorepo, Express + Vite, TypeScript config
  • Database schema + auto-migration system (knex + umzug)
  • Loading bar progress system saat download aset Vite
  • Auth system (JWT + Puter OAuth flow)
  • Seed database tanaman Indonesia (100+ tanaman)
  • API endpoints: /plants, /prices, /auth
S2
Sprint 2 — Minggu 3–4
RAG System & AI Chat
  • Document parser: PDF, DOCX, XLSX, URL scraper
  • Chunking + embedding pipeline via Puter AI
  • Vector storage (sqlite-vec untuk dev, pgvector untuk prod)
  • RAG retrieval + Puter.ai.chat() integration
  • Chat UI dengan Framer Motion, message history
  • Cache layer untuk AI responses populer
S3
Sprint 3 — Minggu 5–6
WhatsApp Bot, Payments & User Features
  • Midtrans integration: charge API, Snap token, webhook handler
  • Escrow flow: escrow hold → release → refund + auto-release 3 hari
  • Dispute system: tabel disputes, admin review flow, resolusi via WA
  • seller_wallets: saldo, pending, withdraw ke bank via Midtrans Transfer
  • Laporan keuangan: /keuangan dashboard + command LAPORAN/SALDO/TARIK via WA
  • Adjustable matching scale: SKALA MATCHING command + preference di DB
  • WhatsApp bot personal dengan Baileys.js (multi-session)
  • User registration + nomor WA verification via OTP
  • Puter.com account linking flow
  • Group bot: event group-participants.update, tabel bot_groups
  • Credit tracking: tabel credit_usage, log tiap AI call
  • Alert & notifikasi harga sistem + dashboard harga komoditas
S4
Sprint 4 — Minggu 7–8
Polish, Testing & Launch
  • Animasi full: page transitions, stagger cards, scroll reveal
  • Mobile optimization, PWA + Service Worker
  • Dark mode implementation
  • Performance audit: Core Web Vitals, bundle size
  • Security audit: rate limiting, input sanitization, CSRF
  • Deploy: Railway/Render (backend) + Cloudflare (CDN)
